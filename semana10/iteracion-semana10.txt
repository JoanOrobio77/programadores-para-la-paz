=== PREGUNTAS CLAVE DEL SPRINT ===

❓ PREGUNTA 1: ¿Qué 2 tareas del backlog harías primero?

RESPUESTA:
1. TASK 1.1: Crear modelo de usuarios en base de datos (5 puntos)
2. TASK 2.1: Crear modelo de eventos (5 puntos)

JUSTIFICACIÓN:
Estas dos tareas son las FUNDAMENTALES del proyecto porque:

✓ SIN USUARIOS: No hay quién interactúe con el sistema
  - Base para autenticación, permisos y trazabilidad
  - Permite identificar quién crea eventos, se registra, participa
  - Esencial para historial, auditoría y reportes

✓ SIN EVENTOS: No hay qué gestionar
  - Es la entidad principal del sistema
  - Todas las otras funcionalidades dependen de eventos existentes
  - Registros, notificaciones, reportes giran alrededor de eventos

IMPACTO EN CADENA:
Task 1.1 → Habilita Task 1.2 (Autenticación) → Permite Task 1.3 (Middleware)
Task 2.1 → Habilita Task 2.2 (Crear) y Task 2.3 (Listar)
Ambas → Permiten Tasks 3.1-3.2 (Registro en eventos)

Son las "piedras angulares" del MVP. Sin estas, todo lo demás es imposible.

---

❓ PREGUNTA 2: ¿Por qué son las más importantes?

RESPUESTA:
Por orden de importancia:

🏆 CRÍTICO (sin estas no hay nada):
1. Usuarios: Identidad y autenticación del sistema
2. Eventos: Núcleo funcional del negocio

🔑 ALTO (dependen de las anteriores):
3. Autenticación: Proteger acceso
4. APIs básicas: Listar y crear eventos

📊 MEDIO:
5. Registro en eventos: Funcionalidad derivada
6. Interfaz: Visualización

📈 BAJO:
7. Notificaciones: Mejora de experiencia
8. Reportes: Análisis posterior

INTERDEPENDENCIAS:
Usuarios ←→ Eventos = Relación 1:N
Usuarios ←→ Registro = Relación M:N (vía Eventos)
Eventos ←→ Notificaciones = Disparador
Registros ←→ Reportes = Base de datos

Sin los dos primeros, el sistema no funciona.
El resto mejora la experiencia pero son secundarios.

---

❓ PREGUNTA 3: ¿Qué significa para ti que una tarea esté "terminada"?

RESPUESTA - DEFINICIÓN DE HECHO (DoD):

Una tarea está TERMINADA cuando cumple TODOS estos criterios:

1️⃣ FUNCIONA CORRECTAMENTE:
   ✓ Código compilado sin errores
   ✓ Lógica implementada según especificación
   ✓ No hay excepciones no capturadas
   ✓ Comportamiento esperado en casos normales

2️⃣ FUE PROBADA ADECUADAMENTE:
   ✓ Tests unitarios escritos y pasando (≥80% cobertura)
   ✓ Tests de integración ejecutados
   ✓ Validación manual de flujos críticos
   ✓ Casos de error probados

3️⃣ QUEDÓ REGISTRADA EN GIT:
   ✓ Commit con mensaje descriptivo
   ✓ PR creado con descripción
   ✓ 2 aprobaciones de code review
   ✓ Sin conflictos pendientes
   ✓ Mergeada a rama develop

4️⃣ DOCUMENTACIÓN COMPLETA:
   ✓ Código comentado en partes complejas
   ✓ README actualizado
   ✓ API documentada (si aplica)
   ✓ Cambios en base de datos documentados

5️⃣ CALIDAD Y PERFORMANCE:
   ✓ Sin warnings de linter
   ✓ Código sigue estándares del proyecto
   ✓ Respuesta < 2 segundos (APIs)
   ✓ Sin deuda técnica

6️⃣ ACEPTACIÓN POR STAKEHOLDER:
   ✓ Product Owner revisó funcionalidad
   ✓ Cumple criterios de aceptación
   ✓ Pasa test de usuario
   ✓ Demostración en Sprint Review

CHECKLIST SIMPLE:
☑ Funciona correctamente
☑ Está probada (tests pasando)
☑ Está en Git (code review aprobado)
☑ Documentada
☑ Sin deuda técnica
☑ Aceptada por Product Owner

CUANDO ALGO NO CUMPLE ESTO:
❌ NO está terminada → Vuelve a backlog
❌ NO se cuenta en velocidad del sprint
❌ Se debe completar antes de siguiente sprint
❌ Se retrasa salida a producción

EJEMPLO:
✓ TERMINADA: Endpoint POST /api/users creado, testado, mergeado, documentado, revisado
❌ NO TERMINADA: Endpoint creado pero sin tests o sin review
❌ NO TERMINADA: Tests pasando pero código no documentado
